# Workout Assistant — Requirement Specification

## 1. Outcomes (Why)

### Problem Statement
Managing workout videos across multiple sources (YouTube, social media, local files) is fragmented. There is no unified system to collect, categorize, and play workout videos, nor to generate structured workout plans from them. This leads to wasted time finding the right video and haphazard training without a plan.

### Objectives
1. **Unified video library** — All workout videos from any source are collected, categorized, stored locally, and playable from a single interface.
2. **Intelligent plan generation** — AI-assisted creation of structured workout plans composed of videos from the library, at any granularity (single session to multi-week programs).
3. **Progress visibility** — Basic tracking of plan completion so the user can see what was done and when.
4. **Any-device access** — Responsive web UI that works equally well on desktop, phone, and tablet.

### Success Metrics
- User can import a video from any supported source and play it in-browser within seconds of import.
- User can generate a workout plan from their library with a single action.
- UI renders and is usable on screens from 320px (phone) to 2560px (desktop) width.
- The system operates fully offline (except AI plan generation which requires API access).

### Stakeholders & Target Users
- **Primary user**: Single individual managing their personal workout routine.
- **No multi-user requirements**: No authentication, no role-based access.

## 2. Capabilities (What)

### Primary Workflows

#### W1: Video Import
User provides a video (file upload or URL from YouTube / direct link / social media). User assigns a custom category and optional metadata (title, description, difficulty, target muscle groups). System downloads (if URL), detects technical metadata (duration, format, size), extracts a thumbnail, and stores the video file on disk organized by category.

#### W2: Video Library Browsing
User browses their video library through the web UI, filtered by category, searchable by title/metadata. User selects a video and plays it in-browser.

#### W3: Workout Plan Generation
User requests a workout plan. System uses an AI backend (OpenAI-compatible API) to generate a plan composed of videos from the user's library. Plan can range from a single session to a multi-week program. User can request regeneration if unsatisfied.

#### W4: Plan Execution & Tracking
User views the current plan, marks individual workout items (or the entire plan) as completed. System logs the completion date. User can view plan history with completion status.

#### W5: Category Management
User creates, renames, and deletes custom categories. Videos are reassigned when categories change.

### System Responsibilities & Boundaries

| Responsibility | Boundary |
|---|---|
| Download videos from URLs | Supported sources: YouTube, direct video URLs, major social media platforms |
| Store video files on local filesystem | Organized by category directory structure |
| AI plan generation | Depends on external OpenAI-compatible API; not available offline |
| In-browser video playback | Stream from local server; no transcoding required if browser supports format |
| Track plan completion | Done/not-done per plan item + completion timestamp |

### Domain Entities (Conceptual)

- **Video**: A workout video file stored locally. Belongs to one category. Has metadata (title, description, difficulty, target muscle groups, duration, format, size, thumbnail).
- **Category**: A user-defined label for organizing videos. User creates, renames, deletes freely.
- **Workout Plan**: A structured collection of videos arranged in a sequence, spanning a defined time range (single session, weekly, multi-week). Generated by AI. Has a creation date.
- **Plan Item**: A single video entry within a plan, with an assigned day/session position. Trackable as completed or not.
- **Plan History**: A log of past and current plans with their completion status.

### Assumptions
- Single user; no concurrent access conflicts.
- Video files are in formats playable by modern browsers (MP4/H.264 preferred).
- The machine has sufficient disk space for video storage (up to ~100 GB).
- AI plan generation requires internet access to reach the configured API endpoint; all other features work offline.
- yt-dlp (or equivalent) can handle YouTube and major social media video extraction.

## 3. Requirements (Shall)

### 3.1 Functional Requirements

#### Video Ingestion
- **FR-01**: The system shall accept video file uploads from the user's device.
- **FR-02**: The system shall accept video URLs from YouTube, direct video links, and social media platforms, and download the video to local storage.
- **FR-03**: The system shall require the user to assign a category to each imported video.
- **FR-04**: The system shall allow the user to optionally provide a title, description, difficulty level, and target muscle groups for each video at import time.
- **FR-05**: The system shall automatically detect and record video duration, file format, and file size upon import.
- **FR-06**: The system shall automatically extract a representative thumbnail from each video.
- **FR-07**: The system shall store video files on the local filesystem, organized by category.
- **FR-08**: The system shall provide import progress feedback (download progress for URLs, upload progress for files).

#### Video Library
- **FR-09**: The system shall display the video library in the web UI with thumbnails, titles, and categories.
- **FR-10**: The system shall allow filtering the video library by category.
- **FR-11**: The system shall allow searching the video library by title and description text.
- **FR-12**: The system shall play videos in-browser using a built-in video player.
- **FR-13**: The system shall allow the user to edit a video's metadata (title, description, category, difficulty, target muscle groups) after import.
- **FR-14**: The system shall allow the user to delete a video, removing both the metadata record and the stored file.

#### Category Management
- **FR-15**: The system shall allow the user to create custom categories.
- **FR-16**: The system shall allow the user to rename existing categories.
- **FR-17**: The system shall allow the user to delete a category, with a prompt to reassign or remove associated videos.

#### Workout Plan Generation
- **FR-18**: The system shall generate workout plans using an AI backend via an OpenAI-compatible API.
- **FR-19**: The system shall allow the user to configure the API endpoint (base URL, API key, model name).
- **FR-20**: The system shall provide the AI with the user's video library (categories, metadata) as context for plan generation.
- **FR-21**: The system shall allow the user to specify plan parameters (e.g., granularity: single session / weekly / multi-week, focus areas, duration preferences).
- **FR-22**: The system shall produce a plan consisting of an ordered list of videos from the user's library, organized by day or session.
- **FR-23**: The system shall allow the user to regenerate a plan (producing a new plan based on the same or modified parameters).

#### Plan Tracking & History
- **FR-24**: The system shall allow the user to mark individual plan items as completed.
- **FR-25**: The system shall record the completion date when a plan item is marked done.
- **FR-26**: The system shall display the current active plan with completion status for each item.
- **FR-27**: The system shall maintain a plan history showing past plans and their completion status.
- **FR-28**: The system shall allow the user to mark an item as un-done (undo completion).

#### User Interface
- **FR-29**: The system shall provide a responsive web UI that is usable on screens from 320px to 2560px width.
- **FR-30**: The system shall provide navigation to the video library, workout plans, plan history, and settings.
- **FR-31**: The system shall provide a settings view for configuring the AI API endpoint (base URL, API key, model name).

### 3.2 Non-Functional Requirements

#### Performance
- **NFR-01**: The video library view shall load within 2 seconds for a library of up to 200 videos.
- **NFR-02**: Video playback shall begin within 3 seconds of the user pressing play.
- **NFR-03**: AI plan generation shall complete within 30 seconds under normal network conditions.

#### Reliability
- **NFR-04**: The system shall not lose video files or metadata if interrupted during plan generation.
- **NFR-05**: If a video download from URL fails, the system shall report the error clearly and allow the user to retry.

#### Usability
- **NFR-06**: The web UI shall be fully functional on the latest versions of Chrome, Firefox, and Safari (desktop and mobile).
- **NFR-07**: The UI shall provide clear visual feedback for all user actions (loading states, success/error messages).
- **NFR-08**: The UI shall be designed for daily use with a polished, non-cluttered aesthetic.

#### Offline
- **NFR-09**: All features except AI plan generation shall function without internet access.
- **NFR-10**: The system shall clearly indicate when AI features are unavailable due to lack of network connectivity.

### 3.3 Data & Domain Definitions

- **Video metadata**: Title (text, required), description (text, optional), category (reference to a Category, required), difficulty (enum: beginner/intermediate/advanced, optional), target muscle groups (text list, optional), duration (number in seconds, auto-detected), format (text, auto-detected), file size (number in bytes, auto-detected), thumbnail (image, auto-extracted), import date (timestamp, auto-set), source URL (text, optional — original URL if applicable).
- **Category**: Name (text, unique, user-defined).
- **Workout Plan**: Name/title (text, AI-generated or user-editable), creation date (timestamp), parameters used for generation (focus areas, duration, granularity), plan items (ordered list).
- **Plan Item**: Video reference, day/session position, completion status (boolean), completion date (timestamp or null).

### 3.4 Observability & Auditability
- **OBS-01**: The system shall log application errors accessible for debugging.
- **OBS-02**: Plan history serves as the audit trail for workout activity.

### 3.5 Operations & Lifecycle
- **OPS-01**: The system shall be deployable on a single machine via a straightforward startup process.
- **OPS-02**: The system shall store all persistent data (video files, metadata database, plan data) in a configurable base directory.
- **OPS-03**: The system shall not require external cloud services for core functionality (video management, playback, tracking).

### 3.6 Edge Cases & Failure Handling
- **EC-01**: If a user imports a URL that cannot be downloaded, the system shall display a clear error message with the reason and offer a retry.
- **EC-02**: If the AI API is unreachable or returns an error during plan generation, the system shall display the error and allow the user to retry.
- **EC-03**: If a video file referenced by a plan is deleted, the plan shall display a clear indicator that the video is missing without crashing.
- **EC-04**: If duplicate videos are imported (same source URL or identical file), the system shall warn the user and ask whether to proceed.
- **EC-05**: If the filesystem runs low on storage during video download, the system shall halt the download and notify the user.

### 3.7 Non-Goals
- Multi-user support, authentication, or authorization.
- Video editing or transcoding.
- Integration with wearable devices or fitness trackers.
- Social features (sharing, comments, feeds).
- Payment, billing, or subscription management.
- Built-in backup/restore functionality (user manages at filesystem level).
- Mobile native apps (web-only).

## Decision Log

| # | Decision | Rationale |
|---|---|---|
| D-01 | Single-user, no authentication | User confirmed personal use only |
| D-02 | Self-hosted, offline-capable | User's hard constraint |
| D-03 | OpenAI-compatible API for AI plans | User requested configurable base URL, API key, model name — this is the standard interface |
| D-04 | Fully custom categories | User preference over predefined lists |
| D-05 | Download all videos locally (including URL-sourced) | User preference for full offline capability |
| D-06 | Plans are video-based only (no free-text exercises) | User preference |
| D-07 | Basic completion tracking (done/not-done + date) | User preference — no sets/reps/weight logging |
| D-08 | No built-in backup/restore | User manages at filesystem level |
| D-09 | Polish from day one | User expects daily-driver quality |
| D-10 | yt-dlp for URL video downloading | Standard tool supporting YouTube + social media; already available on system |
